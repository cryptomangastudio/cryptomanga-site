# マンガ・WEBTOON原作の実務フォーマット

CryptoManga Studiosの中核業務(シナリオ制作・編集・制作進行)に直結する領域。絵で見せる媒体である前提を常に忘れないこと。**セリフとナレーションで説明しすぎない**のが最大の原則。

## 1. 制作フロー全体像

```
プロット(あらすじ) → 箱書き(話数構成) → ネーム(コマ割り台本) → 作画 → (レタリング)
```

- **プロット**:あらすじ。誰が読んでも話の筋が分かる文章。
- **箱書き**:話数ごとに何が起きるかを箇条書きで並べたもの。連載の設計図(テンプレは `story-structure.md` 参照)。
- **ネーム**:実際のコマ割り・構図・セリフ位置まで指定した「マンガの下書き」。原作者・シナリオ担当がここまで書くケースと、作画側がプロット/台本からネームを起こすケースがある。依頼者がどちらを求めているか確認する。
- 作画者に渡す前提のテキストシナリオは「**台本形式**」で書くのが一般的(後述)。

## 2. 台本形式の書き方

作画者・編集が読んで絵に起こせる粒度で書く。以下の要素を明示する:

- **ページ / コマ番号**:「P1-1」「P1-2」のように、ページ番号とコマ番号をセットで振る。
- **柱(ト書き)**:場所・時間・状況などの情報を簡潔に。「(教室、放課後、夕陽が差し込む)」のように。
- **セリフ**:話者名を明記し、セリフ本文を書く。**フキダシ1つに収まる長さ**を意識する(目安:1フキダシ25〜30字前後まで。長台詞は複数フキダシに割る)。
- **効果音・モノローグ・心の声**:通常のセリフと区別して表記する(SE:、M:(モノローグ)、心:(心の声)など。作品内で表記を統一する)。
- **構図・カメラ指示(必要な場合)**:「主人公アップ、目元のみ」「見開き、俯瞰で群衆」など、演出上重要な指示のみ書く。作画の裁量に委ねてよい部分まで細かく指定しすぎない。

### 台本形式のサンプル

```
■第1話「◯◯」 P1

P1-1(大ゴマ)
柱:廃墟と化した街、夜明け前。逆光のシルエットで少女がひとり立つ。
M(ユイ):世界が終わった日、私はまだ生きていた。

P1-2
柱:少女=ユイの足元。割れたスマホの画面に家族写真。
ユイ:「……こんな景色、知ってる」

P1-3(ユイの顔アップ、目元のみ)
ユイ:「——また、やり直せって言うの?」
SE:ゴウ……(風の音)
```

## 3. コマ割り・ページ演出の考え方(紙・冊子想定)

- **見開き2ページを1つの視覚単位として考える**。ページをめくった瞬間に何を見せるかで衝撃・驚き・情報開示のタイミングを設計する(**ページターン効果**)。重要な反転・登場・決め絵は必ず「めくった直後」=偶数ページ側(左開き作品なら逆)に置く。台本にも「※めくり」と明記する。
- **コマの大きさ=時間の長さ・重要度**。大ゴマはインパクトのある瞬間や間を表す。小さいコマを連続させるとテンポが速く感じられる。
- **1ページのコマ数の目安は3〜6**。7コマ以上が続くと窮屈になり、読み飛ばされる。情報を詰めたい話ほど、削る勇気が要る。
- **視線誘導**:右上から左下に読む(日本語マンガの基本)。コマの配置・セリフの位置がこの流れに逆らわないように指定する。
- **1ページの中で「間」を作る**:セリフを詰め込みすぎない。無言のコマ(表情だけ、背景だけ、手元だけ)を意図的に挟むことで、感情の余韻や緊張を作る。感情の山場ほどセリフを減らし、絵に語らせる。

## 4. WEBTOON(縦スクロール)特有の作法

紙マンガとは演出のロジックが根本的に異なる。**見開きの概念がなく、縦に連続するスクロールが基本**。

- **「スクロールスピード」で演出する**。長い縦コマ+余白で緊張や落下感を出す、逆に密度を上げてテンポを速くする、など。**余白は「無」ではなく「間」の演出資源**。
- **1画面(スマホの1スクロール分)ごとに情報を区切る**。次にスクロールしたくなる「めくりポイント」を意図的に作る。台本上でも「――ここで大きく余白、スクロール先で正体の顔を見せる」のように明記すると作画者に伝わりやすい(紙のページターン効果に相当)。
- **横長の構図が使いにくい**。画面幅が狭いため、大人数の横並び・広いパノラマは分割するか縦構図に変換する前提で書く。
- **カラー前提**であることが多く、色や光の演出(回想はセピア、危険は赤、など)も台本の演出指示に含めてよい。
- **話数(エピソード)の切れ目=最大の引き**。話の最後のコマは必ず「続きが気になる」状態で終える。引きの型は `story-structure.md` の5類型を使い分ける。

## 5. 掴み(第1話・各話冒頭)の設計

- 第1話の最初の見開き/最初のスクロール1〜2画面で「これは何のジャンルの、誰の話か」を読者に予感させる。説明よりも**絵になる状況**を優先する(例:設定を語るより、いきなり事件のただ中から始める)。
- セリフでの説明的な自己紹介・世界観説明(「そう、私はこの学園で…」のような独白の長広舌)は基本的に避ける。行動・状況・関係性の中で自然に情報を出す。
- 各話冒頭は「前話の引きの回収」から入るのが基本だが、毎回同じ入り方だと単調になる。ときどき別視点・時間差・肩透かし(緊張→ギャグで外す)を混ぜる。

## 6. ネーム・台本のセルフチェック(納品前に見る)

- [ ] セリフだけ読んで話が「全部」分かってしまわないか(→絵の仕事が残っているか。全部分かるなら説明過多)
- [ ] 逆に、絵の指定なしでは意味が取れないセリフがないか(作画者が構図を誤解する余地)
- [ ] 1フキダシが長すぎるコマはないか
- [ ] 山場の「絵になる瞬間」が1話に最低1つあるか。それはめくり/スクロールの快感位置に置かれているか
- [ ] 無言のコマ(間)がゼロになっていないか
- [ ] キャラの位置関係(左右・距離)がシーン内で矛盾なく追えるか(会話の切り返しで左右が入れ替わる事故)
- [ ] 引きのコマは台本の指定通りの絵で成立するか(文章では盛り上がるが絵にすると地味、がないか)

## 7. 依頼を受けたときの確認事項

原作シナリオ・ネームの依頼を受けたら、以下が不明な場合は文脈から妥当な仮定を明示するか、確認する:

- 媒体(紙連載/WEBTOON/どちらでも展開予定)
- ジャンル・対象読者層(→ `genre-playbook.md`)
- 何を納品するか(プロットのみ/箱書きまで/台本まで/ネーム構成案まで)
- 1話あたりのページ数・尺の目安(紙:連載1話16〜24p、読切32〜48pが典型。WEBTOON:1話50〜80コマ程度が典型)
- 表記ルール(SE/M/心 の記法、キャラ名表記)に既存の社内・作品内基準があるか
