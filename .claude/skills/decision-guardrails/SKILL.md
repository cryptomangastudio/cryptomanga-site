# 意思決定ガードレール(評価ファースト)

CryptoManga Studios の事業(マンガ制作スタジオ本体、cryptobot、affiforge/rakuaffi等のアフィリエイトサイト)で、
ユーザー・売上・資金に影響する変更を提案・実行する前に、このチェックを必ず先に通す。

元ネタは成田悠介氏のAI意思決定論(評価ファースト / 反事実評価 / 単一指標を疑う / 現実制約を先に組み込む /
悪化回避を成果とする / 倫理を計算に入れる / 限定導入)。ここではそれをCryptoManga Studiosの実務に翻訳する。

## 発動したら、実行前に次の5項目を埋める(コードを書く/configを変える前に、まず言語化する)

1. **何を良くしたいか、を一文で書く**
   - 例: 「解約率を下げたい」ではなく「解約率を"月内リピート購入なしユーザー"基準で下げたい」まで具体化する。

2. **測る指標を複数決める(単一指標で判断しない)**
   - 主指標1つ + 副指標2つ以上(悪化してないか監視する指標)を先に決める。
   - 例: cryptobotの戦略変更 → 収益率だけでなく最大ドローダウン・勝率・取引回数もセットで見る。
   - 例: affiforge/rakuaffiのLP変更 → クリック率だけでなく成約率・離脱率・クレーム/返品もセットで見る。
   - 例: マンガのおすすめ/レコメンド施策 → クリック率だけでなく最後まで読了したか・次話への継続率も見る。

3. **本番投入の前に、過去データや小規模検証で先に採点できないか考える**
   - cryptobotなら `backtest.py`(ウォークフォワード + DSR)と `promote.py`(ペーパー→実弾昇格ゲート)を必ず経由する。
     このリポジトリのbotは既にこの思想(段階投入・過学習ゲート)で設計されているので、それを壊す変更(ゲートのスキップ、
     `CRYPTOBOT_LIVE` の安易な有効化など)を提案しない。
   - アフィリエイトサイトやLPなら、全面公開の前に一部トラフィックだけ、または既存版と並行できないか検討する。
   - 過去データが無い/小さすぎる場合は、それを正直に明記した上で「だからこそ小さく始める」と結論づける。

4. **現実の制約を先に洗い出す(理想論で作ってから後付けしない)**
   - 使える予算・工数・締切、法規制(資金決済法・特商法・景表法・税務など)、絶対にやってはいけないこと
     (例: cryptobotの現物のみ制約、レバレッジ禁止、二重ロックの解除条件)を最初に列挙する。

5. **悪化したら気づいて止められる仕組みを先に用意する**
   - 停止ライン/ロールバック手順を、実行前に決めておく(「〇〇が△△を下回ったら中止・元に戻す」)。
   - cryptobotなら既存の日次損失上限・最大ドローダウン全停止・`data/HALTED` の仕組みを使う。壊さない。
   - 新機能や施策は、いきなり全体展開せず、まず小さい範囲(一部ページ/一部ユーザー/一部期間)で試す。

## 出力フォーマット

上記5項目を埋めたら、実装や実行に進む前に短くまとめて提示する:

```
■ 目的:
■ 指標(主/副):
■ 事前検証方法:
■ 制約:
■ 悪化時の停止ライン:
```

その後、ユーザーの明示的な承認を経てから本番相当の変更(live化、config変更、公開、送信など)を行う。
評価の仕組みを作ること自体に時間をかけすぎて、小さく試すべき場面で動けなくならないよう、
この5項目は「重厚な稟議書」ではなく数行のメモで済ませる。過剰な儀式化はしない。
