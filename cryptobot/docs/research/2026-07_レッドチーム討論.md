# レッドチーム討論: 「このbotはまだどこで負けるか」

実施: 2026-07-13 / 手法: 5視点の攻撃者(相場・運用事故・コスト税・統計・人間)がコードを実読して攻撃25件を提出 → 審判が優先順位に統合。

## 攻撃で判明した最重要の欠陥クラスタ(→ 全て実装済み)

### 優先度1: 出口封鎖 —「DD超過=最も売るべき局面」で売りが構造的に不可能だった ✅修正済み
- halt中は売り注文まで却下されていた(「売りは制限しない」という自らの方針と実装が矛盾)
- MAクロスのSELLは「クロスの瞬間」しか出ず、未約定・再起動・halt中に取り逃すと出口が永久消失
- 損切りが未実装で、代替の出口もなかった
- **修正**: halt中も売りは常時許可 / SELLをレベル判定化(fast<slowの間ずっと売り。自己修復する)/ 取得単価比-10%のハードストップ追加 / 買いに0.1%のヒステリシス

### 優先度2: 再起動でブレーカーが武装解除 ✅修正済み
- 日次/週次損失・連敗・月次ガバナー・ピーク資産が全てメモリのみで、再起動一発で消えていた
- 特にピーク資産は「暴落後の資産」で再アンカーされ、HALTED削除→再起動のたびに-15%ずつ切り下がるラチェットが成立していた
- **修正**: `data/risk_state.json` に永続化。状態ファイルがなくても最低「予算額」をピークとみなす保守側フォールバック

### 優先度3: メイカー執行の二重発注 ✅修正済み
- キャンセル失敗(レート制限時)後も注文が板に生きたまま新しい指値を重ねる経路があり、同一意図の注文が最大6枚並び得た
- **修正**: キャンセル完了を確認するまで絶対に再指値しない / 例外パスに線形バックオフ

### 優先度4: paper/live帳簿の混在・実残高との乖離 ✅修正済み
- liveに切り替えるとペーパーの建玉を「実在する」と誤認し、幻の建玉を永久に売り続けるデッドロック
- オーナーの手動売買でも同じ乖離が起きる(急落時に最も起きやすい)
- **修正**: liveモードは帳簿・停止・リスク状態のパスを自動分離(trades_live.csv等)/ 売却前に実残高と突合し、乖離時は実残高分のみ売却+通知

### 優先度5: 出口の非対称エスカレーション ✅修正済み
- 滝落ち相場でpost-only売りが板を追いかけ続け、出口が時間単位で滑っていた
- **修正**: 売りが未約定でシグナル価格から2%滑ったらテイカー成行で確定(入口は従来どおり見送り=非対称)/ 売り残りがある間は1時間待たず60秒周期で再試行

### 優先度6: 二重起動 ✅修正済み
- start.batのダブルクリックで2つのbotが同じ帳簿に書き、リプレイ不能なCSVを作る
- **修正**: ローカルポート占有方式のプロセスロック(プロセス死亡でOSが自動解放)

### その他の実装済み修正
- DCAハードフロア: 価格が200日MA比-25%を割ったら積立自体を停止(浅い押し目と構造的ベアの区別)
- DCA傾斜(dca_tilt)は未検証の平均回帰ベットのため**既定OFF**に変更
- ケリー推定: 直近30回のローリング窓+リターン率ベースに変更。全勝は「無制限」ではなく判定保留。負ケリーでの買い停止時は通知(サイレント永久停止の解消)
- ウォークフォワード: 無取引区間を合格票に数えない / データ不足時は未判定でFAIL

## 受容したリスク(過剰装備は複雑性という新たなリスクを生む)

- レンジ相場のホイップソー: 損失カウンタ永続化により損失速度は予算の数%/週に有界。戦略チューニングの深追いは過学習ゲートの思想と矛盾
- 終値ベースATRの過小評価: max_order_jpy(予算の10%)の絶対上限とハードストップで実害は有界。True Range化は今後の磨き込み
- ペーパー執行の構造的楽観: バックテスト出力で明示警告済み。最終防衛は「最小額の実弾で乖離測定」という段階昇格
- 税率概算の粗さ: 予算10万円の現実的な損益では影響が小さい。年末にreport.pyの総平均法の列を必ず見る運用で受容
- 実効コスト台帳の自動フィードバック: 自己参照的な挙動変化を持ち込みやすい。月1回人間が見て手で直す運用

## 作らないと決めたもの

- 買い側のテイカーフォールバック(エントリーのチェイスは高値掴みの逆選択を悪化させるだけ)
- DSRの全面改修(10万円運用に自己相関補正付き検定装置は工数対効果が逆転)
- ケリー負停止時のプローブモード(最小額の観察買いは「期待値が負でも買う」ことに他ならない)
